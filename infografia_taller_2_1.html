Descripción del Negocio — InpeCable

InpeCable es una empresa proveedora de servicios de telecomunicaciones especializada en televisión por cable e internet. Sus clientes contratan planes de servicio instalados directamente en domicilios o locales comerciales. La empresa gestiona de forma centralizada su cartera de clientes, catálogo de planes, contratos activos, facturación mensual y tickets de soporte técnico.

Los procedimientos almacenados fueron diseñados para automatizar los procesos operativos críticos del Datacenter, reduciendo el tráfico de red, eliminando errores humanos y garantizando la integridad referencial de los datos.

Procedimientos Almacenados Desarrollados

Procedimiento 01
SP_Generar_Factura
IN p_id_contrato
Automatiza la emisión de facturas. Verifica que el contrato exista y esté activo, obtiene el precio del plan asociado y registra la factura con estado "Pendiente" y vencimiento a 15 días.
SELECT COUNT(*) INTO v_count
FROM CONTRATO
WHERE Estado = 'Activo';

INSERT INTO FACTURA (...)
VALUES (v_nro, SYSDATE,
  SYSDATE+15, v_precio,
  'Pendiente', p_id);
COMMIT;
-- EXCEPTION: ROLLBACK
Procedimiento 02
SP_Registrar_Ticket
IN p_dni_cliente IN p_descripcion
Centraliza la lógica de inserción de tickets de soporte técnico. Valida previamente la existencia del cliente (integridad referencial) antes de registrar el ticket con estado "Abierto".
SELECT COUNT(*) INTO v_existe
FROM CLIENTE
WHERE DNI = p_dni;

IF v_existe > 0 THEN
  INSERT INTO TICKET (...)
  VALUES (v_id, SYSDATE,
    p_desc, 'Abierto', p_dni);
  COMMIT;
ELSE RAISE_APPLICATION_ERROR;
Procedimiento 03
SP_Consultar_Deuda
IN p_dni_cliente OUT p_deuda_total
Calcula la deuda acumulada del cliente procesando las sumas directamente en el servidor. Usa un parámetro OUT para retornar el total de facturas pendientes sin necesidad de un SELECT externo.
SELECT COALESCE(
  SUM(f.Monto_Total), 0)
INTO p_deuda_total
FROM FACTURA f
JOIN CONTRATO c
  ON f.ID_Contrato=c.ID
WHERE c.DNI = p_dni
  AND Estado = 'Pendiente';

Técnicas de Programación Aplicadas

📥
Parámetros IN / OUT
Los procedimientos reciben datos mediante parámetros IN (DNI, ID de contrato) y retornan cálculos mediante parámetros OUT (deuda total acumulada), eliminando la necesidad de consultas SELECT externas.
🔀
Control de Flujo IF/THEN
Se emplean estructuras condicionales IF ... THEN ... ELSE para garantizar validaciones cruzadas: verificar estado activo del contrato y existencia del cliente antes de ejecutar inserciones críticas.
⚡
Manejo de Errores EXCEPTION
Bloques EXCEPTION con WHEN OTHERS y RAISE_APPLICATION_ERROR capturan fallos inesperados y ejecutan ROLLBACK automático para proteger la integridad del Datacenter.
🧪

Ejecución y Pruebas

· Prueba de Facturación Exitosa ·
EXEC SP_Generar_Factura('CON-001');
✓ Inserción exitosa de nueva factura asociada al plan del contrato correspondiente. Estado registrado como "Pendiente" con vencimiento a 15 días.
· Prueba de Validación de Error ·
EXEC SP_Registrar_Ticket('00000000', 'Falla');
✗ Sistema bloqueó la acción correctamente. Mensaje: "ORA-20002: El cliente no existe". Integridad referencial garantizada.
· Ejemplo de Bloque Anónimo ·
SP_Consultar_Deuda('12345678', v_deuda);
✓ Retorna la deuda acumulada del cliente mediante parámetro OUT. Output: "La deuda total es: S/ [monto]"
· Decisión de Diseño Principal ·
-- Validación antes de COMMIT
✓ Toda inserción crítica realiza validaciones cruzadas antes del COMMIT. Las reglas de negocio quedan documentadas directamente en el motor de base de datos.
